Software & Operating Systems – Notes

Software Development Life Cycle (SDLC)

Most software projects follow a structured life cycle called the Software Development Life Cycle. While specific models differ, most include similar stages.

Analysis Stage

Stakeholders describe what they want the final system to achieve. This information is used to define the problem and system requirements.

  • Identify inputs, outputs, and stored data
  • Assess current solutions
  • Define functional and non-functional requirements
Example:
A school wants a new attendance system. Requirements include student ID input, daily attendance reports, secure access for staff, and long-term data storage.

Design Stage

The structure of the system is planned before coding begins.

  • Input/output formats
  • User interface layout and accessibility
  • Security and access levels
  • Hardware and compatibility
  • Test plans
Example:
Designing a login screen, defining password rules, and planning test cases for valid and invalid logins.

Development Stage

The system is split into self-contained modules which can be developed in parallel.

Example:
One team codes the database module, another builds the user interface, while another handles authentication.

Testing Stage

The software is tested against the test plan to identify errors.

  • Alpha testing – in-house testing by developers
  • Beta testing – testing by end users
  • White-box testing – tests based on code structure
  • Black-box testing – tests based on inputs/outputs only
Example:
Black-box testing checks that entering an invalid password always produces an error message, regardless of how the code works internally.

Implementation Stage

The completed software is installed and made available to users.

Example:
Installing a new payroll system on company servers and training staff to use it.

Evaluation Stage

The system is reviewed against the original requirements.

  • Reliability
  • Robustness
  • Maintainability
  • Portability
Example:
Checking whether the attendance system reduces teacher workload and produces accurate reports.

Maintenance Stage

Updates are released to fix bugs, patch security issues, or improve functionality.

Example:
A software update fixes a login vulnerability and adds a password reset feature.

Waterfall Model

A linear model where each stage must be completed before the next begins.

  • Clearly documented
  • Easy to manage
  • Inflexible to change
Example:
Developing accounting software with fixed legal requirements.

TELOS Feasibility Study (Waterfall)

  • Technical – is the technology available?
  • Economic – can it be afforded?
  • Legal – is it lawful?
  • Operational – can it be used effectively?
  • Scheduling – can it be completed on time?
Example:
Checking whether a hospital can legally and financially deploy a new patient-record system.

Agile Methodologies

Agile development focuses on flexibility and regular user feedback.

  • Iterative development
  • Early working prototypes
  • High user involvement
  • Less documentation
Example:
A mobile app released with basic features, then updated every few weeks based on user feedback.

Extreme Programming (XP)

An agile methodology using paired programming and constant user input.

  • Two programmers per task
  • User stories define requirements
  • Produces frequent working versions
Example:
Two developers work together on the same code, one writing while the other reviews in real time.

Spiral Model

Designed for large, high-risk projects. Focuses heavily on risk analysis.

  • Identify risks early
  • Terminate project if risks are too high
  • Expensive but thorough
Example:
Developing safety-critical aircraft control software.

Rapid Application Development (RAD)

Uses quick, repeated prototypes to refine requirements.

  • Fast development
  • High user involvement
  • Weaker documentation
Example:
Building a small business website where the client is unsure what features they want initially.

Writing and Following Algorithms

An algorithm is a step-by-step set of instructions used to solve a problem.

  • Clearly defined inputs and outputs
  • Handles invalid inputs
  • Always terminates
  • Well documented and commented
Example:
An algorithm that calculates average exam scores must reject negative marks and divide by the correct number of students.

Comparison of Programming Methodologies

Method Merits Drawbacks Best Use
Waterfall Clear structure, strong documentation Inflexible, little user input Static, low-risk projects
Agile Flexible, high-quality output Poor documentation Small–medium projects with changing requirements
XP Very high code quality High cost, requires teamwork Usability-focused systems
Spiral Excellent risk control Very expensive Large, high-risk projects
RAD Fast development Lower documentation quality Short time-frame projects
