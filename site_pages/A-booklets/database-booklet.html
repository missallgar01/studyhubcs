Databases – Notes

Relational Databases

A relational database stores data in multiple tables. Each table represents a different entity, such as a customer, product, or doctor.

Tables are linked together using keys, allowing data to be shared without duplication.

Example:
A hospital database may contain a Doctor table and a Patient table. These are linked using a DoctorID so each patient can be associated with a doctor.

Entities and Attributes

An entity is something about which data is stored. Attributes are properties describing that entity.

Example: 
Car(CarID, Age, Price)

Here, Car is the entity and CarID, Age, and Price are attributes.

Flat File Databases

A flat file database stores all data in a single table. It is usually based on one entity.

  • Simple to create
  • Hard to maintain
  • High data redundancy
Example:
CarIDAgePrice
Car15 years£1500
Car22 years£2400

Primary Keys

A primary key uniquely identifies each record in a table.

  • Must be unique
  • Must not be null
Example:
In the Car table, CarID is the primary key because no two cars share the same ID.

Foreign Keys

A foreign key links two tables together. It is a primary key in one table that appears in another table.

Example:
DoctorID is the primary key in the Doctor table and a foreign key in the Patient table.

Secondary Keys

A secondary key is used to speed up searching. It does not uniquely identify records.

Example:
Patients may be searched by surname rather than PatientID, so surname is indexed as a secondary key.

Entity Relationship Modelling (ERM)

Entity relationship models show how tables are linked.

  • One-to-one – one record links to one record
  • One-to-many – one record links to many
  • Many-to-many – many records link to many
Example:
One customer can place many orders (one-to-many). Students and courses form a many-to-many relationship.

Normalisation

Normalisation is the process of designing a database to reduce redundancy and improve data integrity.

  • Remove duplicate data
  • Ensure consistent data
  • Allow safe insertion and deletion

Database Normalisation

1NF 2NF 3NF
  • Every table must have a primary key
  • All field names must be unique
  • Values in fields must be from the same domain
  • Values must be atomic (one data item per field)
  • No two records may be identical
  • Must already be in 1NF
  • Remove any partial dependencies
  • Fix any many-to-many relationships created as a result
  • Must already be in 2NF
  • Remove any transitive dependencies
  • Ensure non-key fields are not dependent on other non-key fields

Indexing

Indexing stores records in a specific order to speed up searches.

  • Primary keys are indexed automatically
  • Secondary keys are indexed for fast searching

Capturing Data

Data must be captured accurately and efficiently.

  • Manual entry
  • OCR – Optical Character Recognition
  • OMR – Optical Mark Recognition
  • MICR – Magnetic Ink Character Recognition
Example:
Banks use MICR to read cheque numbers, but the amount is entered manually.

Selecting and Managing Data

Not all collected data is needed. Selecting relevant data reduces processing overhead.

Example:
A speed camera only records vehicles exceeding the speed limit, ignoring all others.

Exchanging Data

Data exchange is the transfer of data between systems.

  • EDI – Electronic Data Interchange
  • No human interaction required
Example:
A supermarket automatically sends stock data to suppliers using EDI.

SQL (Structured Query Language)

SQL is a declarative language used to manage relational databases.

SELECT, FROM, WHERE, ORDER BY

SELECT MovieTitle, DatePublished
FROM Movie
WHERE DatePublished BETWEEN '2000-01-01' AND '2005-12-31'
ORDER BY DatePublished DESC;

JOIN

JOIN combines data from multiple tables using a common attribute. 

SELECT Movie.MovieTitle, Director.DirectorName
FROM Movie
JOIN Director
ON Movie.DirectorName = Director.DirectorName;

CREATE, ALTER, INSERT, UPDATE, DELETE

CREATE TABLE Student (
  StudentID INT PRIMARY KEY,
  Name VARCHAR(30) NOT NULL
);
      
INSERT INTO Student VALUES (1, 'Alex');
      
UPDATE Student SET Name='Alexa' WHERE StudentID=1;
      
DELETE FROM Student WHERE StudentID=1;

Referential Integrity

Referential integrity ensures linked data remains consistent.

Example:
A customer record cannot be deleted if active orders depend on it.

Transaction Processing and ACID

  • Atomicity – all or nothing
  • Consistency – maintains database rules
  • Isolation – transactions don’t interfere
  • Durability – changes persist after failure
Example:
A bank transfer must either fully complete or not occur at all.

Record Locking and Deadlock

Record locking prevents simultaneous updates.

Deadlock Example:
User A locks record 1 and waits for record 2. User B locks record 2 and waits for record 1. Neither can proceed.

Redundancy

Redundancy stores multiple copies of data in different locations.

Example:
Cloud backups ensure data can be recovered after hardware failure.

Question 1

Rosa has recently opened a new leisure centre. Currently she has 150 members, however, she is estimating that this will increase to around 10 000 members.

Rosa will store members' details in a database. She will also store gym class bookings and appointments with fitness instructors.

Rosa is considering using either a flat file database or a relational database.

Discuss the benefits and drawbacks of flat file databases and relational databases and which one would be the most suitable for Rosa.

You should include the following in your answer:

  • The features of flat file databases and relational databases
  • The benefits and drawbacks of each type of database
  • Which type of database would be the most suitable for Rosa

Flat file database:

  • Stores all data in a single table or file
  • Easy to set up and suitable for small amounts of data
  • Can contain redundant (repeated) data
  • Harder to manage and expand as data grows
  • Searching can be slower with large datasets

Relational database:

  • Stores data across multiple related tables
  • Uses primary keys and foreign keys to link data
  • Reduces data duplication
  • Easier to maintain, update, and expand
  • More secure and efficient for large datasets

Conclusion: A relational database is the most suitable for Rosa because the number of members is expected to grow significantly, and it will efficiently manage bookings and appointments while maintaining data integrity.

4(a). Question

A relational database supports ACID transactions. ACID stands for Atomicity, Consistency, Isolation and Durability.

(i) Describe what is meant by a transaction being durable. [2]

(ii) Give one way that durability can be achieved for a completed transaction. [1]

(iii) Explain how record locking can be used to ensure that the ACID principle of isolation is achieved when carrying out multiple transactions. [3]

(iv) Give one disadvantage of using record locking. [1]

(i) Durable means:

  • Committed transaction data is not lost…
  • …even in case of power/system failure.

(ii) One way to achieve durability:

  • Store completed transactions in secondary storage (not just RAM/cache).

(iii) Record locking for isolation:

  • Concurrent transactions should have the same outcome as if they were completed sequentially.
  • Record locking allows only one user/process to access/modify a record at a time.
  • So data being used/modified by one transaction can’t be modified/read in a way that causes interference.

(iv) One disadvantage:

  • Can cause delays (users wait for access), or can cause deadlock.

Question 6

6(a). A hotel uses a computer system to keep track of room bookings. The hotel staff are able to query a database to discover which rooms are booked or which rooms are free.

The hotel’s computer network uses a client-server model.

i. Describe what is meant by the term ‘client-server’ in this context. [3]

ii. Give two advantages of client-server compared to peer-to-peer. [2]

(b). The hotel’s network uses multiple switches. Explain the purpose of a network switch. [3]

(c).* The hotel is concerned about the security of its computer network. Discuss the threats which potentially exist to the hotel’s computer network and how these threats could be eliminated or reduced. [9]

(d). The hotel stores data about rooms, customers and bookings in a database. Each customer can book multiple rooms and each room can be booked multiple times.

i. Draw an Entity Relationship Diagram for this database. [4]

ii. Define what is meant by the term ‘foreign key’, giving one example of where a foreign key would be used in the hotel booking database. [3]

iii. Describe two different ways that hashing could be used in this database. [4]

(e). The hotel booking database enforces referential integrity. Explain what is meant by the term ‘referential integrity’ and how this could potentially be broken. [2]

6(a)(i) Client-server (3 marks):

  • Client computers connect to a server.
  • The server provides access to a resource/service.
  • Here: hotel staff use client machines to query the database hosted on the server.

6(a)(ii) Any two advantages (2 marks):

  • Easier to manage users/access centrally.
  • Easier to back up centrally.
  • Easier to keep data secure centrally.
  • Technicians can remotely install/monitor.
  • (Also commonly credited) only one point of failure.

6(b) Purpose of a switch (3 marks):

  • Connects computers/devices together on a LAN.
  • Receives packets and uses the destination MAC address in the header.
  • Forwards frames out of the correct port to the intended device (not broadcast to all like a hub).

6(c)* Threats + mitigation (indicative points for a high-mark answer):

  • Malware/viruses can damage systems/data → anti-malware, updates, least privilege.
  • Spyware/keyloggers can capture credentials → endpoint protection, staff training, MFA.
  • Phishing steals data via fake messages → staff education, filters, verification processes.
  • DoS/DDoS can disrupt bookings → rate limiting, firewalls, DDoS protection services.
  • Use firewalls, patching, device restrictions (e.g. USB), segmented/restricted Wi-Fi.
  • Balance security measures with customer experience (e.g. guest IT access).

6(d)(i) ERD (4 marks) — textual equivalent:

  • Entities: Customer, Room, Booking (singular names).
  • Links: Customer → Booking is one-to-many; Room → Booking is one-to-many.
  • No direct Customer–Room link (Booking acts as the link).

6(d)(ii) Foreign key (3 marks):

  • Definition: A field that links to the (primary) key in a second table.
  • Example: CustomerID (or RoomID) stored in the Booking table.

6(d)(iii) Hashing uses (4 marks):

  • Security: hash passwords in the database so they can’t be read if stolen.
  • Direct access/indexing: use a hash of an index/key to quickly locate Customer/Room/Booking records.

6(e) Referential integrity (2 marks):

  • Foreign keys must reference an existing/valid primary key (relationships remain consistent).
  • It can be broken if a primary key is deleted/updated without cascading, leaving orphan foreign keys.
